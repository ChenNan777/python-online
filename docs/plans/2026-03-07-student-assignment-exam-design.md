# 学生作业考试模式设计

**目标**

- 在不影响练习模式的前提下，为挑战页接入考试模式学生作业流程。
- 考试模式统一接入 `assignment-info`、`saveCode`、`submit`、历史代码查询接口。
- 考试模式的题面、初始代码、测试数据、地图路网、测试点位、定位分析场景与评分结果都以接口数据为准。

## 背景与问题

- 当前练习模式是纯本地交互，题目数据来自 `src/pages/ChallengePage/challenges.ts`。
- 当前挑战页 `src/pages/ChallengePage/index.tsx` 同时承载编辑器、运行、调试、测试与地图可视化逻辑。
- 如果继续在单页内堆叠 `isPracticeRoute` 分支，后续考试模式接入后端真实数据、保存、提交、评分与历史代码时，页面逻辑会快速膨胀。
- 如果完整复制一份考试页面，又会造成两套编辑器和调试工作区维护成本过高。

## 设计结论

- 拆分为两个容器页：`PracticeChallengePage` 与 `ExamChallengePage`。
- 抽取共享工作区：`ChallengeWorkspace`。
- 练习模式维持纯本地交互；考试模式改为纯后端数据驱动。
- 共享工作区不感知数据来源，只消费统一的工作区模型。

## 页面与模块边界

### PracticeChallengePage

- 负责从本地静态题库装配练习数据。
- 保留现有练习模式全部能力。
- 不调用任何学生作业接口。
- 路径规划继续从本地资源读取路网；定位分析继续使用本地场景数据。

### ExamChallengePage

- 负责从后端装配考试题目、作业上下文、历史代码与提交流程。
- 接入以下接口：
  - `GET /StudentOperationCode/assignment-info`
  - `POST /StudentOperationCode/saveCode`
  - `POST /StudentOperationCode/submit`
  - `GET /StudentOperationCode/list/student/{taskId}/{memberId}`
- 考试模式下，题面、代码、测试数据、地图路网、测试点位、定位场景、附加信息与评分结果均来自后端。

### ChallengeWorkspace

- 负责共享 UI 与运行时能力：
  - 编辑器
  - 运行 / 调试控制
  - 输出与测试结果展示
  - 地图与定位可视化面板
- 不直接调用考试接口。
- 不直接依赖本地静态题库或后端原始 DTO。

### Adapter / Hooks

- 通过页面级 adapter / hook 负责把本地数据或后端 DTO 转换为统一工作区模型。
- 考试模式下单独维护保存状态、提交状态、历史代码状态和评分结果状态。
- `workType` 固定映射：
  - 定位分析：`4`
  - 路径规划：`5`

## 统一工作区模型

- 工作区只消费一份标准化后的 challenge model，至少包含：
  - 题目标题与描述
  - 初始代码 / 当前代码
  - 测试数据
  - challenge 类型（定位分析 / 路径规划）
  - 地图或定位可视化场景数据
  - 调试能力开关
  - 结果展示所需数据

这样练习容器与考试容器的差异只体现在“如何装配数据”，而不是“如何渲染工作区”。

## 路径规划场景设计

- 练习模式：
  - 路网、测试点位和调试数据来自本地资源。
  - 保留当前本地交互能力，例如自定义起终点调试。
- 考试模式：
  - 路网、测试点位、起终点、附加信息等全部来自后端。
  - 是否允许拖点调试等能力由考试配置决定，不能默认继承练习模式的所有本地调试行为。

## 定位分析场景设计

- 统一引入“定位场景数据模型”，不再把 `stations` 和 `measurements` 作为两段独立来源散落使用。
- 合并规则：通过 `stationId` 与 `station.id` 进行关联，形成统一场景结构。
- 练习模式：本地生成原始 `stations` 与 `measurements` 后，在 adapter 层完成按 ID 合并，再提供给工作区与 Python 上下文。
- 考试模式：
  - 先通过 `assignment-info` 获取 `targetId`
  - 再通过后续预留接口按 `targetId` 查询定位分析真实场景
  - 虽然该接口暂未提供，但前端先保留两段式装配能力
  - 返回值同样要转换为统一定位场景模型

## 历史代码设计

- 考试模式进入页面后，按当前 `taskId + memberId` 获取历史代码列表。
- 历史代码支持：
  - 列表查看
  - 代码预览
  - 一键恢复到编辑器
- 恢复只回填 `sourceCode`，不自动提交。
- 若当前代码存在未保存修改，恢复前需要确认，避免无提示覆盖。

## 保存与提交流程

### 保存

- 仅考试模式启用自动保存。
- 用户编辑代码后做防抖保存，避免每次输入都触发请求。
- 保存失败时保留未保存状态，并允许后续继续重试。

### 提交

- 点击提交时先做兜底保存，再走统一提交接口。
- 提交时按 challenge 类型附带正确的 `workType`。
- 提交失败不清空编辑器，也不误报成功。
- 提交成功后刷新必要的作业态与结果展示。

## 错误处理

- `assignment-info` 拉取失败：提示错误，但尽量保留页面基础可见性。
- 历史代码拉取失败：不阻塞考试主流程，但提示历史功能不可用。
- 场景数据拉取失败：工作区不进入错误数据态，明确提示当前考试场景缺失。
- 自动保存失败：展示保存失败状态，允许继续编辑。
- 提交失败：明确提示，不丢失当前代码。

## 验收标准

- 练习模式功能保持不变，且不调用学生作业相关接口。
- 考试模式通过统一学生作业接口完成当前作业加载、自动保存、提交与历史代码查询。
- 考试模式地图相关数据全部来自后端，而不是本地固定资源。
- 考试模式定位分析场景走统一模型，`stations` 与 `measurements` 按 ID 合并。
- 历史代码支持查看与恢复到编辑器。
- 共享工作区内部不出现大量练习 / 考试分支判断。

## 暂未实现但需预留的点

- 定位分析按 `targetId` 获取真实场景的接口尚未提供。
- 前端先预留考试定位场景二段式装配能力，避免后续再次重构工作区。
