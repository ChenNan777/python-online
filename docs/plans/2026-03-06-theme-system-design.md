# 主题系统设计

## 背景

当前项目整体是亮色风格，仓库内已经存在少量主题相关痕迹，但还没有完整的全局主题系统。

- `src/styles/modules/theme.scss` 目前只包含少量基础 token。
- `src/components/EditorX/types.ts` 已支持 `"light" | "dark"`。
- 多处 Monaco 使用了写死的 `theme="vs"`，说明主题能力尚未统一收口。

需求目标是：在保留当前亮色主题的前提下，新增一个偏科技风的暗黑主题，并支持主题切换；同时为后续增加更多主题预留扩展能力。

## 目标

- 支持全局主题切换。
- 第一阶段提供两个主题：`light` 与 `dark-tech`。
- 暗黑主题视觉方向为科技风，但以可读性与层次为优先。
- 主题选择需要持久化，刷新后保持。
- 后续新增第三套、第四套主题时，不需要重构现有架构。

## 推荐方案

采用“主题系统骨架 + 当前落地亮色和暗黑科技风两套主题”的方案。

核心思想：

- 用一个统一的全局主题状态表示当前主题。
- 在根节点挂载 `data-theme`，让全局样式基于语义 token 切换。
- 对 Ant Design、Monaco、业务面板这三类内容分别做主题适配。
- 第一阶段优先覆盖高可见页面和公共容器，避免一次性重写全部样式。

不推荐只做“把页面改暗”而不抽象主题层，因为这会让后续新增主题时付出更高重构成本。

## 架构设计

### 1. 主题状态层

新增一个轻量全局主题状态模块，推荐放在 `src/store/useThemeStore.ts`。

建议状态最小化为：

- `themeId`
- `setTheme(themeId)`
- `loadFromStorage()`

可选补充：

- `availableThemes`

但第一阶段不必把过多展示信息塞进 store，避免过度设计。

### 2. DOM 挂载方式

应用启动时将当前主题写入：

- `document.documentElement.dataset.theme = themeId`

这样全局 CSS / SCSS 可通过：

- `:root[data-theme="light"]`
- `:root[data-theme="dark-tech"]`

进行主题 token 覆盖。

### 3. 持久化策略

主题值写入 `localStorage`。

建议键名独立，不与鉴权数据混用，例如：

- `app_theme`

读取失败或值非法时回退到 `light`。

## 视觉与样式设计

### 1. Token 分层

将 `src/styles/modules/theme.scss` 从当前的少量变量扩展为语义化 token 层。

建议至少覆盖：

- `--bg-app`
- `--bg-panel`
- `--bg-elevated`
- `--text-primary`
- `--text-secondary`
- `--border-default`
- `--accent-primary`
- `--accent-soft`
- `--shadow-panel`
- `--glow-accent`
- `--radius-sm`
- `--radius-md`

优先使用语义 token，而不是 `--blue-500` 这类纯色值命名，便于未来主题扩展。

### 2. 亮色主题

亮色主题原则上保持当前体验，不做大范围视觉重做。

第一阶段只做：

- 把已有亮色样式映射到新的 token
- 修正个别在暗黑适配中需要统一收口的公共容器

### 3. 暗黑科技风主题

暗黑主题采用深蓝黑底色，不用纯黑，以保留层次。

推荐特征：

- 主背景为深蓝黑
- 面板层级比背景稍亮
- 强调色使用青蓝或电光蓝
- 主按钮、选中态、聚焦边框有轻微发光效果
- 大面积区域不过度使用霓虹光效，避免疲劳和可读性下降

科技风主要落在以下元素：

- 面板边框
- 输入框 / 选择器聚焦态
- 主要按钮
- 激活标签
- 编辑器外层容器

## 组件与页面覆盖范围

第一阶段建议优先覆盖：

- 全局页面背景与基础文字
- 登录页
- Dashboard 页面
- Debugger / Challenge 页面主体容器
- 常用公共面板组件
- Editor 容器

对业务页面的原则是：

- 先覆盖容器和基础层级
- 再覆盖高频交互元素
- 最后再处理视觉细节

## 第三方能力适配

### 1. Ant Design

在根部加入 `ConfigProvider`，让 Ant Design token 跟随全局主题切换。

第一阶段至少让以下内容视觉统一：

- Button
- Input
- Form
- Card
- Modal
- Tabs
- message / feedback 类组件

### 2. Monaco Editor

当前多处页面直接写死 `theme="vs"`，需要统一改成由主题状态驱动。

目标行为：

- `light -> vs`
- `dark-tech -> vs-dark`

若后续要做更强科技风的编辑器配色，再在 Monaco 自定义主题层继续扩展。

### 3. 地图与可视化区域

第一阶段优先改容器、说明层、图例和浮层，不强制切换底图风格。

原因：

- 底图深色化往往需要额外资源或单独配置
- 当前需求重点是主题系统，而不是地图视觉重做

## 交互设计

### 1. 切换入口

增加一个主题切换入口，建议放在全局可见区域，例如：

- Dashboard 页头
- 顶部工具栏
- 调试页上方操作区

第一阶段使用简单切换控件即可：

- Switch
- Segmented
- Dropdown

推荐用能体现“未来多主题”的控件，例如下拉或分段选择，而不是只做一个硬编码暗黑开关。

### 2. 默认行为

- 默认进入 `light`
- 用户切换后立即生效
- 刷新页面后沿用上次选择

## 错误处理与回退

- `localStorage` 读取失败时回退到 `light`
- 未识别的 `themeId` 回退到 `light`
- 若某些第三方组件暂未完全适配，优先保证功能正确，不阻断主流程

## 分阶段实施建议

### 第一阶段

- 建立主题 store
- 建立 `data-theme` 注入逻辑
- 扩展全局 token
- 打通 Ant Design 与 Monaco 主题切换
- 覆盖高可见页面与公共容器

### 第二阶段

- 补齐零散组件的硬编码颜色
- 统一更多业务页面视觉细节
- 视需要增加更完整的暗黑地图 / 图表适配

### 第三阶段

- 增加更多主题
- 增加主题元数据与主题预览
- 若有需要，增加系统主题跟随能力

## 验证方案

至少验证以下内容：

- 主题切换后刷新仍保持
- `light` 与 `dark-tech` 可正常切换
- Ant Design 常见组件颜色正确
- Monaco 跟随主题变化
- 登录页、Dashboard、Debugger、Challenge 主体可读性正常
- `npm run lint` 通过
- `npx tsc -p tsconfig.app.json --noEmit` 通过
- 若改动涉及构建链路或运行时，`npm run build` 通过

## 不做项

第一阶段暂不包含：

- 多套暗黑子主题
- 自动跟随系统主题
- 深色地图资源重做
- 全站逐像素级视觉重构

## 结论

本次应优先建设“可扩展的主题基础设施”，而不是只做一次性的暗黑样式覆盖。

这样既能尽快交付亮 / 暗切换，也能保证未来继续增加主题时，变更集中在 token、主题定义和少量适配层，而不会再次拆改整站样式结构。
